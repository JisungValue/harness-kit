# Coding Conventions Project

## 목적

이 문서는 공통 `coding_guidelines_policy.md` 위에 프로젝트 전용 coding convention만 추가한다.

## 시작 방법

- 언어별 convention 초안이 필요하면 `bootstrap/language_conventions/`에서 해당 언어 템플릿을 골라 이 문서에 병합한다.
- 단일 언어 프로젝트면 필요한 항목만 이 문서에 직접 복사해 시작한다.
- 다중 언어 프로젝트면 공통 규칙은 이 문서에 두고, 언어별 상세 규칙은 별도 문서로 분리한 뒤 여기서 참조해도 된다.

## 언어별 convention 연결

- 현재 프로젝트가 실제로 사용하는 언어와 런타임 기준을 먼저 적는다.
- bootstrap 템플릿을 복사했다면 어떤 언어 템플릿을 기준으로 병합했는지 적는다.
- 언어별 상세 규칙을 별도 문서로 분리했다면 정확한 문서 경로를 여기서 참조한다.
- 현재 프로젝트에서 실제로 적용하는 규칙만 남기고, 사용하지 않는 언어/런타임 규칙은 그대로 끌고 오지 않는다.
- `[프로젝트 결정 필요]` 표시는 남길 수 있지만, 현재 Phase 2에서 바로 영향을 주는 항목은 구현 전에 프로젝트 기준으로 확정한다.

## Phase 2 구현/감사 체크리스트용 요약

- 현재 프로젝트의 활성 언어/런타임: `[프로젝트 결정 필요]`
- bootstrap 출처 또는 기준 언어 문서: `[프로젝트 결정 필요]`
- 현재 프로젝트에서 우선 적용하는 핵심 규칙 범주: `[naming / modeling / error handling / concurrency / collections / testing / interop 등]`
- 현재 프로젝트의 주요 금지 패턴: `[프로젝트 결정 필요]`
- 현재 Phase 2에서 미해결로 둘 수 없는 언어별 결정 항목: `[프로젝트 결정 필요]`
- formatter/linter/type checker/test의 실행 명령, 필수 여부, 적용 시점은 이 문서가 아니라 `docs/project/standards/quality_gate_profile.md`에 둔다.

## Phase 2에서 확인해야 할 것

- 구현자는 현재 작업과 직접 관련 있는 언어별 규칙을 이 문서에서 먼저 식별한다.
- 감사자는 구현이 이 문서의 활성 규칙과 금지 패턴을 실제로 따르는지 확인한다.
- 현재 변경과 직접 관련 없는 언어 규칙까지 형식적으로 모두 점검하지 않는다.
- 현재 변경에 영향을 주는 항목이 아직 `[프로젝트 결정 필요]` 상태면, 구현/감사 전에 기준을 먼저 확정한다.

## 여기에 둘 내용 예시

- 프레임워크 특화 관례
- DTO postfix 또는 suffix 규칙
- 트랜잭션 경계 규칙
- 프로젝트 전용 에러 타입 규칙
- 프로젝트 전용 로깅 필드 규칙

## 금지

- core에 이미 있는 규칙을 중복해서 다시 쓰지 않는다.
- 여러 프로젝트에 공통인 규칙을 project 전용 규칙처럼 두지 않는다.
- 현재 프로젝트에서 쓰지 않는 언어 규칙을 audit noise로 남기지 않는다.
- 언어별 핵심 금지 패턴을 적지 않은 채 장문 설명만 남기지 않는다.
