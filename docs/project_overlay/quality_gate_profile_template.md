# Quality Gate Profile

## 목적

이 문서는 프로젝트에서 실제로 돌려야 하는 품질 게이트 명령과 실패 처리 기준을 정의한다.

## 책임 경계

- formatter, linter, type checker, test, 조건부 architecture rule 같은 실행 게이트는 이 문서에서 정의한다.
- 테스트 환경 준비, coverage 기준, 통합 테스트 대상, 외부 의존성 기동 방식은 `testing_profile.md`에서 정의한다.

## 최소 포함 항목

- formatter 실행 명령
- linter 실행 명령
- type checker 실행 명령
- test 실행 명령
- 필요한 경우 architecture rule 검사 명령
- 각 명령의 필수 여부
- 각 명령의 적용 시점
- 실패 시 처리 기준

## 권장 섹션 예시

### Formatter

- 명령: `[프로젝트 결정 필요]`
- 필수 여부: `[필수/선택]`
- 적용 시점: `[저장 시 / 커밋 전 / Phase 2 종료 전 / Phase 3 종료 전 / Phase 4 기록 시 / CI 전용]`
- 실패 처리 기준: `[프로젝트 결정 필요]`

### Linter

- 명령: `[프로젝트 결정 필요]`
- 필수 여부: `[필수/선택]`
- 적용 시점: `[프로젝트 결정 필요]`
- 실패 처리 기준: `[프로젝트 결정 필요]`
- code hygiene를 linter 또는 별도 check로 다룬다면 어떤 항목(import, dead code, debug 흔적 등)을 이 게이트가 책임지는지 적는다.

### Type Checker

- 명령: `[프로젝트 결정 필요]`
- 필수 여부: `[필수/선택]`
- 적용 시점: `[프로젝트 결정 필요]`
- 실패 처리 기준: `[프로젝트 결정 필요]`

### Test

- 명령: `[프로젝트 결정 필요]`
- 필수 여부: `[필수/선택]`
- 적용 시점: `[프로젝트 결정 필요]`
- 실패 처리 기준: `[프로젝트 결정 필요]`
- 세부 테스트 범위와 환경은 `testing_profile.md`를 참조한다.

### Architecture Rule

- 명령: `[프로젝트 결정 필요]`
- 필수 여부: `[없으면 생략 가능]`
- 적용 시점: `[프로젝트 결정 필요]`
- 실패 처리 기준: `[프로젝트 결정 필요]`
- design quality를 architecture rule로 일부 자동화한다면 어떤 경계, 계층, 의존성 규칙을 검사하는지 적는다.
- 자동화되지 않는 design quality 판단은 `implementation_notes.md`를 수동 기록 위치로 사용한다고 명시한다.

## 작성 원칙

- 특정 도구를 core 기본값으로 강제하지 않는다.
- 실제로 팀이 반복 실행하는 명령만 적는다.
- 명령이 없으면 왜 없는지 또는 어떤 수동 검증으로 대체하는지 남긴다.
- `필수 여부`와 `적용 시점`을 함께 적어, 현재 phase에서 바로 실행해야 하는지 later phase 또는 CI에서만 실행해도 되는지 구분한다.
- `testing_profile.md`와 중복 설명하지 않고, 테스트 세부 기준은 해당 문서를 참조한다.
