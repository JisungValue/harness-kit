# Commit Rule

## 목적

이 문서는 프로젝트에서 커밋을 만들 때 지켜야 할 일반 규칙과 팀별 커스터마이징 지점을 함께 정의한다.

## 사용 방법

- 아래 규칙은 여러 프로젝트에서 공통으로 적용 가능한 기본값이다.
- `[팀 결정 필요]`로 표시한 항목은 팀 정책에 맞게 값을 채우거나 문구를 수정한다.
- 팀 결정이 끝나지 않은 항목은 임시 기본값으로 두되, 프로젝트 시작 단계에서 우선 확정한다.

## 기본 원칙

- 알려진 compile, type, build, test 실패 상태로 커밋하지 않는다.
- 커밋은 가능한 한 원자적으로 유지한다.
- 원자적 커밋 원칙보다 깨지지 않는 상태 유지를 우선한다.
- 현재 커밋 의도와 직접 관련 없는 대규모 정리나 리팩터링은 분리한다.

## 커밋 전 최소 점검 항목

- 현재 커밋 범위에서 compile 오류가 없는가
- 현재 커밋 범위에서 type 오류가 없는가
- 현재 커밋 범위에서 build 실패가 없는가
- 현재 커밋 범위에서 필수 테스트가 통과하는가
- 실패를 인지한 상태에서 임시 우회 코드를 남기지 않았는가

## 기본 실행 프로파일 예시

- `[팀 결정 필요]` 필수 검사 명령
  - 예: `lint`, `typecheck`, `build`, `test`
- `[팀 결정 필요]` 필수 테스트 범위
  - 예: 변경 파일 연관 단위 테스트 + 핵심 통합 테스트
- `[팀 결정 필요]` 실패 허용 범위
  - 예: flaky 테스트 재시도 횟수, CI 일시 장애 시 대응 절차

## 커밋 메시지 규칙 기본안

- 기본 형식은 `<type>(<scope>): <subject>`를 권장한다. `scope`는 선택이다.
- `type` 기본값 예시는 `feat`, `fix`, `docs`, `refactor`, `test`, `chore`, `perf`, `ci`, `build`, `revert`다.
- 제목(`subject`)은 50자 이하로 작성한다.
- 제목은 동사로 시작하고 현재형/명령형으로 작성한다.
- 제목 끝에 마침표를 쓰지 않는다.
- 본문은 필요 시 작성하되, 변경 내용 나열보다 변경 이유와 trade-off를 우선 설명한다.
- 관련 이슈가 있으면 footer에 `Refs: #번호` 또는 `Closes: #번호`를 남긴다.
- 호환성 깨짐이 있으면 footer에 `BREAKING CHANGE:`를 명시한다.

## 커밋 메시지 예시

- `feat(auth): add token refresh endpoint`
- `fix(cache): prevent stale profile read`
- `docs(harness): clarify phase 2 audit gate`

## 원자성 운영 기준

- 하나의 커밋은 하나의 의도된 변경 단위를 설명할 수 있어야 한다.
- 여러 의도가 섞이면 커밋을 분리한다.
- 단, 분리 시 저장소가 깨지면 먼저 깨지지 않는 상태로 재구성한 뒤 커밋한다.

## 금지 항목

- known failing 상태를 인지한 채로 "나중에 고친다"는 전제로 커밋하지 않는다.
- 임시 디버깅 코드, 의미 없는 로그, 주석 처리된 잔여 코드를 그대로 커밋하지 않는다.
- 현재 변경과 무관한 대규모 코드 청소를 같은 커밋에 혼합하지 않는다.

## 예외 처리

- 긴급 장애 대응처럼 즉시 커밋이 필요한 경우에도, 깨지지 않는 상태 원칙은 유지한다.
- 예외를 적용한 경우 `변경 이유`, `임시 조치`, `후속 복구 계획`을 PR 또는 작업 로그에 남긴다.

## 팀별 커스터마이징 항목

- `[팀 결정 필요]` 커밋 메시지 형식(Conventional Commit 유지 여부, 허용 type, scope 사용 규칙)
- `[팀 결정 필요]` 제목 길이 제한(기본 50자)과 제목 시작 동사 목록
- `[팀 결정 필요]` 브랜치 전략(main 보호 규칙, squash/rebase/merge 정책)
- `[팀 결정 필요]` CI 필수 체크 목록과 merge 차단 기준
- `[팀 결정 필요]` 문서/마이그레이션 변경 시 추가 점검 규칙

## 비범위

- 템플릿 기본안을 넘는 팀 고유 prefix, scope, ticket, release naming 규칙은 강제하지 않는다.
