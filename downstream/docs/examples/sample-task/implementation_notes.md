## 구현 중 결정 사항

- repo-local 근거: `docs/harness/common/design_quality_policy.md`, `docs/harness/common/performance_policy.md`, `docs/examples/sample-task/validation_report.md`를 기준으로 예시 기록 형식을 정리했다.
- repo에 없어 문서화/승인 대상으로 넘긴 결정: sample task는 실제 프로젝트가 아니므로 project-specific 품질 게이트 명령과 concrete architecture 결정은 실제 도입 프로젝트 overlay에서 확정해야 한다.

- performance 검토 기록 예시:
  - 대상 경로: 대량 사용자 목록을 export용 DTO로 변환하는 배치 함수
  - 입력 규모 또는 제약: 최대 50,000건 배치, 메모리 급증 없이 완료해야 함
  - 관찰 증상 또는 우려 근거: 초안 구현이 `filter -> map -> group -> map` 형태라 중간 컬렉션이 세 번 생성되고, export payload 복사가 한 번 더 발생한다.
  - 검토한 항목:
    - 시간복잡도: 같은 컬렉션을 여러 번 순회한다.
    - 메모리: 중간 컬렉션과 큰 payload 복사가 연속 발생한다.
    - 반복 호출 또는 I/O: 외부 I/O는 없지만 동일 필드 가공 계산이 반복된다.
  - 대안:
    - 대안 1: 선언형 단계 분리를 그대로 유지한다.
    - 대안 2: 단일 순회 기반으로 줄이되 함수 둘로 분리해 책임은 유지한다.
  - 추천안: 대안 2
  - 근거:
    - 최대 배치 크기 기준으로 중간 컬렉션과 payload 복사 비용이 누적된다.
    - 단일 순회로 줄여도 함수 경계를 유지해 읽기 가능성을 크게 잃지 않는다.
  - 검증 방법 또는 후속 검증 계획: 샘플 10건과 대량 50,000건 데이터셋으로 변환 함수 실행 시간을 비교하고, peak memory는 로컬 profiler 또는 runtime metric으로 확인한다.
  - trade-off:
    - 얻는 것: 중간 컬렉션 감소, payload 복사 감소, 메모리 안정성 개선
    - 잃는 것: 단계별 선언형 표현은 일부 줄어든다
  - 지금 하지 않는 대안의 이유: 선언형 단계 분리는 읽기 쉽지만 현재 배치 규모 기준으로 비용이 설명 가능하게 커진다.

- design quality 판단 기록 예시:
  - 상황: 외부 응답 파싱, 유효성 검사, 저장 호출을 한 함수에서 처리하는 구조를 어떻게 정리할지 판단이 필요했다.
  - 함수 분리 판정 질문 적용 결과:
    - 하나의 동사로 자연스럽게 요약되는가: 아니오
    - 검증, 계산, 외부 호출, 저장이 섞여 있는가: 예
    - 같은 함수 안에 독립 변경 이유가 두 가지 이상 있는가: 예
  - 대안:
    - 대안 1: interface/facade를 추가해 parsing과 persistence를 분리한다.
    - 대안 2: 같은 파일 안에서 검증/변환/저장 단계를 함수로 분리하고 이름을 정리한다.
  - 추천안: 대안 2
  - 근거:
    - 현재 구현체가 하나뿐이라 interface 도입 근거가 약하다.
    - 함수 분리만으로도 책임과 테스트 포인트가 충분히 분명해진다.
    - 현재 TASK 범위를 넘는 구조 재설계를 피할 수 있다.
  - trade-off:
    - 얻는 것: 읽기 쉬움, 테스트 가능성, 책임 분리
    - 잃는 것: 미래 교체 가능성을 위한 추상화는 아직 없다
  - 지금 하지 않는 대안의 이유: interface/facade 추가는 현재 요구보다 앞선 추상화라 이번 TASK에서는 도입하지 않는다.

- design quality vs performance 충돌 기록 예시:
  - 상황: 큰 컬렉션을 처리하는 코드에서 loop를 여러 단계로 나누면 읽기 쉬워지지만, 검증용 샘플 데이터 기준으로 중간 컬렉션 생성이 두 번 발생해 메모리 사용량이 커졌다.
  - 충돌 발생 판정 질문 적용 결과:
    - 관찰된 병목 근거가 있는가: 예
    - 더 단순한 설계안이 현재 성능 목표를 충족하지 못하는가: 예
    - 더 빠른 구조를 택하면 설계 품질이 악화되는가: 예
  - 대안:
    - 대안 1: 단계별 중간 컬렉션을 유지한 채 더 읽기 쉬운 구조를 유지한다.
    - 대안 2: 한 번의 순회로 줄이는 쪽으로 코드를 합치되, 함수 이름과 주석 없이도 읽히는 수준으로만 단순화한다.
  - 추천안: 대안 2
  - 근거:
    - 현재 샘플 기준으로 중간 컬렉션 2회 생성이 확인됐다.
    - 한 번의 순회로 줄여도 책임을 함수 둘로 분리해 읽기 가능성을 유지할 수 있다.
  - 검증 방법 또는 후속 검증 계획: 샘플 데이터와 대량 데이터셋 각각에서 처리 시간과 peak memory를 다시 확인하고, 실제 프로젝트에서는 batch size 상한 기준으로 재검증한다.
  - trade-off:
    - 얻는 것: 메모리 사용 감소, 불필요한 중간 컬렉션 제거
    - 잃는 것: 단계별 선언적 표현은 조금 줄어든다
  - 지금 하지 않는 대안의 이유: 더 읽기 좋은 형태지만 현재 관찰된 메모리 비용을 정당화할 정도로 단순하지 않다.
